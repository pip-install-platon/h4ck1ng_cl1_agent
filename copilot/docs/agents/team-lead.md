# Team-Lead (оркестратор)

## Роль

- Диалог с оператором (TTY `session`, позже LLM — узко и с контролем выхода в контракты).
- **Порядок Kill Chain**, выбор следующего модуля и формулировка `directive`.
- Сбор **`knowledge`** из графа по `lane` / engagement; подстановка **`skillPack`** через SkillResolver.
- Приём **`AgentResult`**, валидация, merge **`graphOps`**, при `failed` / `blocked` — **replan** (новая задача, другой модуль, уточнение у пользователя).

## Не делает

- Не гоняет сканеры и не пишет PoC сам по умолчанию — это специализированные модули.
- Не хранит «истину» в свободном тексте чата — только в графе и валидированных контрактах.

## Стратегия и соответствия

- **Маппинг CVE ↔ находка ↔ тактика (ATT&CK / методология)** — зона совместной ответственности Team-Lead + **Vuln** + **Report**: воркеры дают **нормализованные факты** и `evidenceRef`, Team-Lead сводит к приоритетам и следующему шагу.
- Выбор «идти в эту CVE или сначала расширять поверхность» — **прерогатива Team-Lead**, с учётом цели пользователя и скоупа.

## Рантайм

Логика оркестрации в `packages/core` (например `TeamLeadRunner`); полноценный LLM-слой — следующий билд. **Отдельный `module` в `AgentTask` для Team-Lead не требуется** — это слой над воркерами.
